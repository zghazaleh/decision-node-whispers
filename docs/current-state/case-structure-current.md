# Case Structure — Current State

How a mission is actually shaped in code. The shipped shape diverges from the idealized model in `docs/architecture/case-structure.md`; this file describes the running code.

## Two parallel catalogues

There are two independent sources of truth keyed on the same `mission-0X` id:

1. **Display catalog** — `src/lib/missions.ts`, exported as `MISSIONS: MissionMeta[]`. Used by `/missions` (`src/routes/missions.tsx`). Fields: `id, number, codename, title, logline, status, route?, duration?, tone?, location?, year?, category?, difficulty?, creator?, version?`. `status` is `"available" | "classified" | "locked"`.
2. **Runtime engine** — `src/lib/missions/registry.ts`, populated from `src/lib/missions/mission-0X/index.ts`. Used by `/api/chat`, `/mission/$id`, `/sound-test`, and the analyzer.

The two are not cross-checked. A mission listed in `MISSIONS` with no engine would render on `/missions` and fail at `/mission/$id` via the `useEffect` redirect to `/missions`. An engine registered without a `MISSIONS` entry would be playable by direct URL but invisible in the case-file grid.

## `MissionEngine` shape

Declared in `src/lib/missions/types.ts`. Every mission folder exports a `MissionEngine`:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | string | Must match the folder/registry key. |
| `systemPrompt` | string | Full prompt sent as `system` to the chat model. Includes the canon block. |
| `opening` | `{ text: string }` | Rendered as the synthetic first assistant message (NOT generated by the model). Must include a `<<chips: …>>` line. |
| `scene` | `{ src, filter?, mood? }` | Background image + CSS filter. |
| `atmosphere` | `MissionAtmosphere?` | Optional FX/audio overrides (see types.ts). |
| `archetypes` | `Record<string, Archetype>` | All authored stances keyed by id. |
| `archetypeIds` | `string[]` | Stable id order, excluding `"unclassified"`. |
| `archetypeMenuForClassifier` | `() => string` | Text menu fed to the Stage A classifier. |
| `getArchetype` | `(id) => Archetype \| null` | Lookup. Returns `null` for `"unclassified"`. |
| `decisionPresets` | `DecisionPreset[]` | Buttons in the Decide modal. Each has `{label, text, archetypeId}`. |
| `canon` | `Record<string, unknown>` | Shape is mission-specific; not validated beyond "object with ≥1 field". |

`Archetype` (from `types.ts`) — `{ id, label, matchHints[], timeline: ArchetypeBeat[], secondOrder: Record<string, string>, tone }`. Each `ArchetypeBeat` is `{ beat, consequence }`.

## Per-mission triad

Each `src/lib/missions/mission-0X/` exports:

- `index.ts` — assembles the `MissionEngine` (system prompt, scene, atmosphere, decision presets, exports the engine).
- `canon.ts` — declares `CANON` (typed `as const`) and `canonGroundTruthBlock()` that formats canon into the textual block appended to `systemPrompt`.
- `outcomes.ts` — declares `ArchetypeId` union, `ARCHETYPES` record, `ARCHETYPE_IDS` derived list, `getArchetype()`, and `archetypeMenuForClassifier()`.

The mission-01 triad is the only one inspected in detail for this doc; mission-02..04 follow the same triad pattern (confirmed by directory listing and mission-02 partial read). Each canon is **shape-specific** — e.g. mission-01's canon has `world, company, decisionWindow, player, characters, objects, history36h, constraints`, while other missions use different keys.

### `mission-01` snapshot

- 5 archetypes: `ship_on_time`, `hold_two_weeks`, `narrow_release`, `indefinite_pause`, `step_down`. Plus a sentinel `"unclassified"` in the type union that is never registered in `ARCHETYPES`.
- Each archetype carries a 6-beat timeline (T+1d → T+1y), a 7-key `secondOrder` map (`orion9, aperture, helios, marcus, amara, board, elena`), and a `tone` string.
- 5 decision presets, one per archetype id.
- Canon includes 5 characters (sarah, marcus, amara, jonas, david), 5 objects (note, memo, laptop, photograph, portfolio), a 9-step `history36h` log, and 6 hard constraints.

## Validation — `src/lib/missions/validation.ts`

Runs once at registration (`registry.ts → register()`):

- Zod schema for `MissionEngine`. Non-empty strings on id, systemPrompt, opening.text, scene.src; archetypes record with ≥1 entry; archetypeIds ≥1; decisionPresets ≥1; canon must be an object with ≥1 top-level field.
- Cross-field checks beyond Zod:
  - Every `archetypeIds[i]` must exist in `archetypes`.
  - Every `archetypes[key].id` must equal its key.
  - Every `decisionPresets[i].archetypeId` must reference an existing archetype.
- On failure: the engine is **not** registered; errors are stored in `VALIDATION_ERRORS[id]` and a single-line `console.error` is logged. `getMissionEngine(id)` returns `null`. `requireMissionEngine(id)` throws.

### Assumptions

- Canon schemas are mission-private — there is no shared schema validating canon fields. A canon refactor that drops `CANON.company.competitor.name` in mission-01 would *not* be caught by validation, but would render as `"undefined"` inside the system prompt via `canonGroundTruthBlock()`.
- The `"unclassified"` id is treated as a runtime fallback, never registered as an archetype, never an option in `decisionPresets`.

### Fragile / implicit behavior

- Registration order (`mission-01 → 02 → 03 → 04`) matters only for which validation errors print first; lookup is by id.
- The chat route silently coerces an unknown `missionId` to a 400, but never re-checks validation errors — a mission with validation errors is treated as "unknown" from the client's perspective.
- The `MissionEngine.scene.src` field comes from a Vite-resolved image import (`@/assets/scene-*.jpg`), so engines fail at build time if the asset path is wrong, not at validation time.

### Risks if changed

- Renaming an archetype id requires editing both `outcomes.ts` (id + key) *and* the matching `decisionPresets[].archetypeId` in `index.ts`. Validation catches the mismatch loudly, but at module load time only — a typo discovered in production would prevent that mission from loading until redeployed.
- Removing the `archetypeMenuForClassifier` export from `outcomes.ts` removes the menu the Stage A classifier reads from; the function is stored as a value on the engine and called at analysis time.
